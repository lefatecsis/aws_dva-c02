Pergunta 1Ignorado
Qual das alternativas a seguir melhor descreve como funciona a criptografia no KMS?
O KMS gera uma nova CMK para cada chamada de Encrypt e criptografa os dados com ela.

Explicação

Incorreta. O KMS não gera uma nova CMK a cada chamada de criptografia. Em vez disso, utiliza uma CMK existente para criptografar os dados. Embora o KMS possa rotacionar automaticamente as chaves para melhorar a segurança, ele não cria uma nova CMK para cada operação de criptografia.
Resposta correta
O KMS gerencia as CMKs e gera chaves de dados que os clientes usam para criptografar localmente seus dados, protegendo as CMKs internamente.

Explicação

Correta. O KMS gerencia as chaves mestras do cliente (CMKs) e gera chaves de dados (data keys) que são usadas pelos clientes para criptografar e descriptografar seus dados localmente. O KMS protege as CMKs internamente e não recebe os dados dos clientes para criptografá-los diretamente.
O KMS envia a CMK para o cliente, que realiza a criptografia e depois apaga a CMK.

Explicação

Incorreta. O KMS nunca envia a chave mestra (CMK) para o cliente. Toda a criptografia e descriptografia são realizadas internamente no KMS, garantindo que a chave nunca seja exposta fora do serviço.
O KMS recebe a CMK do cliente a cada chamada de Encrypt e criptografa os dados com ela.

Explicação

Incorreta. Embora seja possível importar uma CMK para o KMS, isso é feito uma única vez. Após a importação, o KMS gerencia a chave e realiza as operações de criptografia e descriptografia. Não é necessário enviar a CMK a cada chamada de criptografia.

💡 GLOSSÁRIO
● KMS (Key Management Service): serviço gerenciado da AWS que cria, gerencia e controla chaves criptográficas usadas para proteger dados.
● CMK (Customer Master Key): chave mestra gerenciada pelo KMS, usada para gerar chaves de dados e realizar operações criptográficas.
● Chave de dados (Data Key): chave gerada pelo KMS para criptografar e descriptografar dados localmente, mantendo a CMK protegida.
● Criptografia local: criptografia de dados realizada no ambiente do cliente utilizando chaves fornecidas pelo KMS, sem enviar os dados para o KMS para criptografia direta.
● Rotação de chaves: prática de substituir periodicamente chaves criptográficas para aumentar a segurança.
● Importação de chave: processo de trazer uma chave externa para ser gerenciada pelo KMS internamente.

🌐 EXPLICAÇÃO GERAL

  • O AWS KMS gerencia as CMKs e gera chaves de dados que são utilizadas pelos clientes para criptografar/descriptografar informações localmente.

  • A CMK nunca sai do KMS, garantindo que as chaves mestras permaneçam protegidas.

  • Essa abordagem combina segurança centralizada (controle das CMKs no KMS) com eficiência operacional (criptografia local pelos clientes).

  • Conceitos incorretos incluem: gerar uma nova CMK a cada operação ou enviar a CMK para o cliente, práticas que não ocorrem por questões de segurança.

🚀 COMO ISSO SE APLICA NA VIDA REAL

  • Uma aplicação serverless que grava dados sensíveis no DynamoDB ou S3 pode usar o KMS para gerar chaves de dados, criptografando os dados localmente antes do armazenamento.

  • Isso garante que a CMK permaneça segura, facilita gerenciamento centralizado e permite rotação automática de chaves.

  • Equipes de CI/CD podem integrar o KMS para proteger segredos e credenciais durante implantações, evitando exposição acidental e garantindo conformidade com políticas de segurança corporativas.

  • Essa prática é essencial em ambientes que exigem alta segurança e conformidade, mantendo dados e chaves protegidos de forma consistente.

Pergunta 2Ignorado
Uma empresa de mídia criou um aplicativo de streaming de vídeo e deseja que seus usuários brasileiros sejam atendidos pelos servidores localizados no Brasil. Usuários de outras regiões do mundo não devem conseguir acessar esses servidores por meio de consultas DNS. Qual política de roteamento do Route 53 atende a esse requisito?
Resposta correta
Geolocation (Geolocalização)

Explicação

Correta. A política Geolocation permite direcionar o tráfego com base na localização geográfica de onde as consultas DNS se originam, possibilitando que apenas usuários brasileiros acessem os servidores no Brasil, enquanto bloqueia ou redireciona usuários de outras regiões.
Latency (Latência)

Explicação

Incorreta. A política Latency roteia o tráfego para a região AWS que oferece a menor latência para o usuário, visando melhorar o desempenho, mas não restringe o acesso por localização geográfica.
Failover (Failover)

Explicação

Incorreta. A política Failover direciona o tráfego para um recurso principal quando ele está saudável e para um recurso secundário quando o principal está indisponível, não controlando o acesso por localização geográfica.
Weighted (Ponderado)

Explicação

Incorreta. A política Weighted permite distribuir o tráfego entre múltiplos recursos em proporções definidas, mas não restringe o acesso com base na localização geográfica dos usuários.

💡 GLOSSÁRIO
● Route 53: serviço de DNS gerenciado da AWS que traduz nomes de domínio em endereços IP e permite roteamento inteligente do tráfego.
● Geolocation (Geolocalização): política de roteamento do Route 53 que direciona tráfego com base na localização geográfica do usuário que faz a consulta DNS.
● Weighted (Ponderado): política que distribui o tráfego entre múltiplos recursos em proporções definidas.
● Failover: política que direciona tráfego para um recurso principal enquanto está saudável e para um secundário em caso de falha.
● Latency (Latência): política que direciona o tráfego para a região AWS com menor latência para o usuário.
● DNS (Domain Name System): sistema que traduz nomes de domínio legíveis para endereços IP numéricos.
● Consulta DNS: pedido enviado para o serviço DNS para resolver um nome de domínio em um endereço IP.

🌐 EXPLICAÇÃO GERAL

  • A política Geolocation do Route 53 permite restringir o acesso a servidores com base na localização geográfica do usuário, identificando a origem da consulta DNS.

  • Com Geolocation, é possível direcionar usuários brasileiros para servidores no Brasil e bloquear ou redirecionar usuários de outras regiões.

  • Outras políticas como Weighted, Failover e Latency não oferecem controle geográfico:

    • Weighted → balanceamento proporcional de tráfego

    • Failover → alta disponibilidade entre primário e secundário

    • Latency → otimização de desempenho com menor latência

🚀 COMO ISSO SE APLICA NA VIDA REAL

  • Em um aplicativo de streaming de vídeo, o desenvolvedor pode garantir que usuários brasileiros sejam atendidos por servidores localizados no Brasil, melhorando a experiência com menor latência e cumprindo requisitos regulatórios locais.

  • Usuários de outras regiões podem ser impedidos de acessar os servidores brasileiros, protegendo recursos e otimizando custos.

  • Essa configuração também é útil para compliance com políticas de soberania de dados, permitindo segmentação geográfica do tráfego sem necessidade de mudanças complexas na aplicação ou infraestrutura.

  • Facilita a manutenção e escalabilidade da aplicação na AWS, mantendo performance e conformidade regulatória.

Pergunta 3Ignorado
Um desenvolvedor foi encarregado de proteger determinados buckets S3 que são compartilhados por uma grande equipe de usuários. Na última vez em que a política de um bucket foi alterada, ele ficou erroneamente disponível para qualquer pessoa, inclusive fora da organização. Qual recurso/serviço ajudará o desenvolvedor a identificar problemas de segurança semelhantes com o mínimo de esforço?
Resposta correta
IAM Access Analyzer

Explicação

Correta. O AWS IAM Access Analyzer ajuda a identificar recursos na sua organização e contas, como buckets Amazon S3 ou funções IAM, que estão compartilhados com entidades externas. Isso permite detectar acessos não intencionais aos seus recursos e dados, reduzindo riscos de segurança.
Recurso Access Advisor no console IAM

Explicação

Incorreta. O Access Advisor informa o último uso de funções IAM para ajudar a identificar funções não utilizadas, auxiliando na remoção segura dessas funções. No entanto, ele não fornece informações sobre entidades fora do IAM, como buckets S3, e portanto não é adequado para este cenário.
S3 Analytics

Explicação

Incorreta. O Amazon S3 Analytics Storage Class Analysis ajuda a analisar padrões de acesso para otimizar a transição de dados entre classes de armazenamento. Ele não é capaz de identificar acessos não intencionais ou problemas de segurança em buckets S3.
S3 Object Lock

Explicação

Incorreta. O S3 Object Lock permite armazenar objetos usando o modelo ""Write Once Read Many"" (WORM), prevenindo exclusões acidentais ou inadequadas de dados. Porém, não é adequado para identificar problemas de acesso ou políticas incorretas em buckets S3.

💡 GLOSSÁRIO
● IAM Access Analyzer: ferramenta da AWS que identifica recursos compartilhados com entidades externas à organização, destacando acessos não intencionais.
● S3 Object Lock: recurso WORM do S3 que previne exclusões ou alterações em objetos, garantindo integridade.
● Access Advisor: mostra o último uso das permissões associadas a usuários ou funções IAM, útil para identificar permissões não utilizadas.
● S3 Analytics: analisa padrões de acesso para otimizar classes de armazenamento, sem foco em segurança.
● Bucket Policy: conjunto de regras que define quem pode acessar um bucket S3 e quais ações são permitidas.
● Zona de confiança: escopo interno versus externo usado para definir limites de análise de acesso.
● WORM (Write Once Read Many): modelo de armazenamento em que dados gravados não podem ser alterados ou excluídos, garantindo integridade.

🌐 EXPLICAÇÃO GERAL

  • O IAM Access Analyzer é a ferramenta indicada para identificar buckets S3 expostos inadvertidamente a entidades externas à organização.

  • Ele analisa políticas e permissões, destacando recursos acessíveis fora da zona de confiança.

  • Outras opções, como S3 Object Lock, Access Advisor, S3 Analytics ou Bucket Policies manuais, não oferecem análise automatizada ou abrangente de acessos externos, tornando o Access Analyzer a escolha ideal para este cenário.

🚀 COMO ISSO SE APLICA NA VIDA REAL

  • Em empresas com múltiplas equipes usando buckets S3 compartilhados, alterações nas políticas podem abrir brechas de segurança.

  • Um engenheiro de segurança pode usar o IAM Access Analyzer para monitorar continuamente esses buckets e identificar qualquer exposição externa.

  • Por exemplo, em pipelines que usam AWS Lambda para processar dados S3, o Access Analyzer garante que apenas funções ou usuários autorizados tenham acesso, prevenindo vazamentos e mantendo conformidade com normas internas e regulatórias.

  • Além disso, ele fornece alertas e relatórios que facilitam ações corretivas rápidas, mantendo o ambiente seguro e controlado.

Pergunta 4Ignorado
Um desenvolvedor está testando filas do Amazon Simple Queue Service (SQS) em um ambiente de desenvolvimento. A fila, juntamente com todo o seu conteúdo, precisa ser excluída após os testes. Qual API do SQS deve ser utilizada para atender a esse requisito?
RemovePermission

Explicação

Incorreta. A operação RemovePermission é usada para revogar permissões específicas definidas na política da fila, identificadas pelo parâmetro Label. Ela não exclui a fila nem suas mensagens, apenas remove permissões associadas.
PurgeQueue

Explicação

Incorreta. A operação PurgeQueue exclui todas as mensagens contidas na fila especificada pelo QueueUrl, mas mantém a fila ativa. Ou seja, a fila permanece disponível para uso, apenas seu conteúdo é removido.
RemoveQueue

Explicação

Incorreta. RemoveQueue não é uma operação válida na API do Amazon SQS e foi incluída apenas como uma alternativa para confundir. A exclusão correta da fila deve ser feita via DeleteQueue.
Resposta correta
DeleteQueue

Explicação

Correta. A operação DeleteQueue exclui a fila especificada pelo parâmetro QueueUrl, independentemente do conteúdo da fila. Ao deletar a fila, todas as mensagens nela contidas deixam de estar disponíveis. O processo de exclusão pode levar até 60 segundos, durante os quais algumas requisições podem ainda ser aceitas, mas após esse período a fila e suas mensagens deixam de existir. Além disso, é necessário aguardar pelo menos 60 segundos antes de criar uma nova fila com o mesmo nome.

💡 GLOSSÁRIO
● DeleteQueue: operação da API do Amazon SQS que exclui a fila especificada e todas as mensagens nela contidas, tornando a fila indisponível.
● PurgeQueue: operação que remove todas as mensagens da fila, mas mantém a fila ativa e disponível para uso.
● RemovePermission: operação que revoga permissões específicas definidas na política de acesso da fila, sem afetar a existência da fila ou suas mensagens.
● QueueUrl: identificador único e obrigatório para referenciar uma fila específica nas chamadas da API do Amazon SQS.
● Amazon Simple Queue Service (SQS): serviço gerenciado de filas de mensagens que permite desacoplar e escalar microsserviços, sistemas distribuídos e aplicações serverless.

🌐 EXPLICAÇÃO GERAL
A alternativa correta é DeleteQueue, pois:

  • Remove completamente a fila e todo o seu conteúdo, atendendo ao requisito de exclusão total após os testes.

  • PurgeQueue apenas limpa as mensagens, mantendo a fila ativa, o que não satisfaz a necessidade de exclusão total.

  • RemovePermission só atua sobre permissões e não impacta a fila em si.

  • RemoveQueue não existe na API do SQS.

Usar DeleteQueue segue as boas práticas da AWS para gerenciamento de recursos temporários em ambientes de desenvolvimento, evitando custos desnecessários e confusões com filas obsoletas.

🚀 COMO ISSO SE APLICA NA VIDA REAL

  • Em desenvolvimento e testes, times podem criar filas SQS para simular fluxos de mensagens entre componentes.

  • Após a validação, é fundamental limpar o ambiente para evitar acúmulo de recursos e garantir testes futuros consistentes.

  • DeleteQueue remove a fila e todas as mensagens de forma definitiva, liberando recursos e evitando interferência em novos testes.

  • Em pipelines de CI/CD, excluir programaticamente filas após cada execução mantém o ambiente limpo e controlado.

  • Equipes que trabalham com arquiteturas serverless (Lambda + SQS) podem usar DeleteQueue para gerenciar filas temporárias criadas dinamicamente durante testes automatizados.

Pergunta 5Ignorado
Uma empresa multinacional acabou de migrar para a AWS Cloud e configurou alertas baseados em previsões no AWS Budgets para gerenciamento de custos. No entanto, nenhum alerta foi recebido, mesmo após quase três semanas da criação da conta e dos orçamentos. Qual poderia ser o problema na configuração do AWS Budgets?
O orçamento baseado em previsão foi criado a partir de uma conta que não possui privilégios suficientes.

Explicação

Incorreta. Se a conta ou o usuário não tivesse privilégios suficientes, o orçamento nem poderia ser criado. Portanto, essa não é a causa do problema.
Resposta correta
A AWS exige aproximadamente 5 semanas de dados de uso para gerar previsões de orçamento.

Explicação

Correta. O AWS Budgets necessita de cerca de 5 semanas de dados históricos de uso para gerar previsões confiáveis. Até que esse período seja alcançado, alertas baseados em previsão não serão disparados.
A conta precisa fazer parte de uma Organização AWS para receber alertas do AWS Budgets.

Explicação

Incorreta. Contas independentes também podem criar orçamentos e receber alertas. Não é obrigatório pertencer a uma Organização AWS para utilizar o AWS Budgets.
O Amazon CloudWatch pode estar indisponível, e por isso os alertas não estão sendo enviados.

Explicação

Incorreta. O Amazon CloudWatch é um serviço totalmente gerenciado pela AWS e raramente fica indisponível. Essa alternativa foi incluída apenas como um fator de distração.

💡 GLOSSÁRIO
● AWS Budgets: serviço que permite definir orçamentos personalizados para monitorar custos e uso na AWS, com possibilidade de configurar alertas.
● Previsão de orçamento: estimativa futura dos custos baseada em análise de dados históricos de uso.
● Amazon CloudWatch: serviço gerenciado da AWS para monitoramento, coleta de métricas e configuração de alarmes.
● AWS Organizations: serviço que permite gerenciar múltiplas contas AWS centralmente, facilitando controle e governança.
● Dados históricos de uso: informações acumuladas sobre consumo de recursos e serviços ao longo do tempo.
● Alertas baseados em previsão: notificações enviadas quando o custo projetado ultrapassa o limite definido no orçamento.
● Privilégios de conta: permissões necessárias para criar, modificar e gerenciar recursos e configurações na AWS.
● Contas independentes: contas AWS que operam isoladamente, sem fazer parte de uma organização.

🌐 EXPLICAÇÃO GERAL
O AWS Budgets precisa de um período mínimo de aproximadamente cinco semanas de dados históricos para gerar previsões confiáveis de custos futuros.

  • Durante as primeiras semanas, alertas baseados em previsão podem não ser disparados, pois ainda não há histórico suficiente para estimativas precisas.

  • Outras causas, como problemas no CloudWatch, falta de privilégios ou ausência de uma organização AWS, não explicam a falta de alertas nesse cenário:

    • CloudWatch é altamente disponível e não depende de histórico.

    • Privilégios insuficientes impediriam a criação do orçamento, não apenas o disparo de alertas.

    • Contas independentes recebem alertas sem necessidade de estar vinculadas a uma organização.

Portanto, a ausência de alertas após três semanas é comportamento esperado.

🚀 APLICAÇÃO NO MUNDO REAL
Um desenvolvedor ou time financeiro que gerencia custos em uma empresa que migrou recentemente para a AWS deve compreender:

  • Ao configurar orçamentos para monitorar gastos com EC2, Lambda ou outros serviços, é necessário aguardar o período inicial de coleta de dados antes que os alertas baseados em previsão funcionem corretamente.

  • Planejar esse tempo de maturação dos dados evita falsas expectativas quanto ao disparo de alertas.

  • Essa prática garante monitoramento financeiro mais preciso e alinhado às boas práticas de governança na nuvem, permitindo melhor controle de custos e decisões de otimização.

Pergunta 6Ignorado
Sua organização global possui uma infraestrutura de TI implantada usando CloudFormation na AWS Cloud. Um funcionário, na região us-east-1, criou uma stack chamada 'Application1' e exportou uma saída com o nome 'ELBDNSName'. Outro funcionário criou uma stack para uma aplicação diferente, 'Application2', na região us-east-2, e também exportou uma saída com o nome 'ELBDNSName'. O primeiro funcionário tentou implantar a stack CloudFormation 'Application1' na região us-east-2, mas ocorreu um erro. Qual é a causa desse erro? (Selecione duas.)
Os valores de saída exportados no CloudFormation devem ter nomes únicos em todas as regiões.

Explicação

Incorreta. Os nomes das saídas exportadas precisam ser únicos apenas dentro da mesma região, não entre diferentes regiões. Portanto, essa afirmação está incorreta.
Seleção correta
Os valores de saída exportados no CloudFormation devem ter nomes únicos dentro de uma única região.

Explicação

Correta. Essa é a regra correta: os nomes exportados devem ser únicos dentro da mesma região para evitar conflitos entre stacks.
Uma stack CloudFormation não pode ser implantada em mais de uma região da AWS usando o mesmo nome de stack.

Explicação

Incorreta. É possível criar stacks com o mesmo nome em diferentes regiões da AWS. O erro relatado não está relacionado ao nome da stack, mas sim ao conflito de nomes de exportações dentro da mesma região.
Seleção correta
Os valores de saída no CloudFormation devem ter nomes únicos dentro de uma única região.

Explicação

Correta. No CloudFormation, os nomes das saídas exportadas devem ser únicos dentro da mesma região para evitar conflitos. Como ambas as stacks exportaram 'ELBDNSName' na região us-east-2, houve um conflito que causou o erro.
Os valores de saída no CloudFormation devem ter nomes únicos em todas as regiões.

Explicação

Incorreta. Não existe restrição para nomes únicos de saídas exportadas entre regiões diferentes, apenas dentro da mesma região.

💡 GLOSSÁRIO
● CloudFormation: serviço da AWS para provisionamento e gerenciamento de recursos de infraestrutura usando templates declarativos em JSON ou YAML.
● Stack: conjunto de recursos AWS criados e gerenciados como uma única unidade pelo CloudFormation.
● Output Values (Valores de saída): informações definidas em uma stack que podem ser exportadas para serem usadas por outras stacks.
● Exported Output Values (Valores de saída exportados): valores de saída disponibilizados para referência cruzada entre diferentes stacks na mesma região.
● Região (Region): área geográfica da AWS onde os recursos são implantados, como us-east-1 ou us-east-2.

🌐 EXPLICAÇÃO GERAL
No CloudFormation, os nomes dos valores de saída exportados devem ser únicos dentro da mesma região para evitar conflitos entre stacks:

  • Duas stacks na mesma região não podem exportar saídas com o mesmo nome; caso contrário, a implantação falhará.

  • Essa restrição não se aplica entre regiões diferentes: stacks em regiões distintas podem usar nomes de saída iguais sem problemas.

  • No exemplo apresentado, a stack Application1 tentou exportar ELBDNSName em us-east-2, onde já existia uma saída com esse nome, causando conflito e erro de implantação.

🚀 COMO ISSO SE APLICA NA VIDA REAL
Desenvolvedores que gerenciam infraestrutura como código precisam garantir unicidade de nomes de saída exportados dentro da mesma região:

  • Ao criar múltiplas aplicações que compartilham recursos, como um Load Balancer, é comum exportar seu DNS para que outras stacks possam referenciá-lo.

  • Se duas stacks na mesma região exportarem o mesmo nome, a implantação falhará, exigindo renomeação da saída para resolver o conflito.

  • Em ambientes multi-região, é possível reutilizar nomes de saída iguais, permitindo padronização de templates e automação de deploys regionais.

Compreender essa regra ajuda a evitar erros de implantação, manter a organização e modularidade das stacks e garantir consistência em ambientes distribuídos globalmente.

Pergunta 7Ignorado
O Amazon Simple Queue Service (SQS) possui um conjunto de APIs para diversas ações suportadas pelo serviço. Como um desenvolvedor associado, quais das seguintes afirmações você identificaria como corretas em relação à API CreateQueue? (Selecione três)
Seleção correta
O valor do tempo de visibilidade para a fila é em segundos, com padrão de 30 segundos

Explicação

Correta. O tempo de visibilidade da fila é configurado em segundos, com valores válidos entre 0 e 43.200 (12 horas). O valor padrão é 30 segundos, que determina o período em que uma mensagem fica invisível após ser recebida por um consumidor.
Seleção correta
A fila de dead-letter (DLQ) de uma fila FIFO deve ser também uma fila FIFO. Já a DLQ de uma fila padrão deve ser também uma fila padrão

Explicação

Correta. A fila de dead-letter de uma fila FIFO deve ser obrigatoriamente uma fila FIFO. Da mesma forma, a fila de dead-letter de uma fila padrão deve ser também uma fila padrão; não é permitido misturar os tipos entre a fila principal e a DLQ.
Seleção correta
Você não pode alterar o tipo da fila após criá-la

Explicação

Correta. O tipo da fila não pode ser alterado após sua criação, ou seja, não é possível converter uma fila padrão em FIFO. Para usar uma fila FIFO, é necessário criar uma nova fila desse tipo ou excluir a fila padrão existente e recriá-la como FIFO.
A API CreateQueue pode ser usada para atualizar os atributos de uma fila existente passando os mesmos parâmetros da criação original.

Explicação

Incorreta. A API CreateQueue é usada exclusivamente para criar uma nova fila. Se você tentar criar uma fila com o mesmo nome de uma existente e passar atributos diferentes, o SQS retorna um erro. Para alterar atributos de uma fila existente, deve-se usar a API SetQueueAttributes.
As tags da fila são case sensitive. Se você adicionar uma nova tag com a mesma chave, mas com letras maiúsculas/minúsculas diferentes, ambas serão mantidas separadamente.

Explicação

Incorreta. As tags da fila no Amazon SQS são case insensitive. Isso significa que chaves como 'env' e 'ENV' são tratadas como iguais. Se uma nova tag for adicionada com a mesma chave (independentemente do uso de maiúsculas/minúsculas), ela sobrescreve a existente.
O tempo, em segundos, pelo qual a entrega de todas as mensagens na fila é atrasada é configurado usando o atributo MessageRetentionPeriod

Explicação

Incorreta. O atraso na entrega das mensagens é configurado pelo atributo DelaySeconds, que define o tempo em segundos para atrasar a entrega de todas as mensagens na fila. O atributo MessageRetentionPeriod controla o tempo, em segundos, que o Amazon SQS mantém uma mensagem antes de descartá-la.

🌐 EXPLICAÇÃO GERAL
A API CreateQueue do Amazon SQS permite configurar diversos atributos críticos durante a criação da fila:

  • Tipo da fila (padrão ou FIFO) é fixo e não pode ser alterado posteriormente, garantindo comportamento consistente.

  • Visibility Timeout controla, em segundos, por quanto tempo uma mensagem fica invisível após ser recebida (padrão de 30 segundos), evitando duplicação de processamento.

  • Filas FIFO devem ter DLQs FIFO, e filas padrão devem ter DLQs padrão, mantendo coerência no tratamento de mensagens com falha.

  • Tags são case insensitive: uma nova tag com chave igual sobrescreve a existente.

  • DelaySeconds define atraso inicial na entrega das mensagens; MessageRetentionPeriod define apenas o tempo de retenção antes da exclusão.

Essas configurações permitem que filas sejam criadas de forma previsível e consistente, evitando problemas de duplicidade ou perda de mensagens.

  • Essa abordagem desacopla backend e apresentação, melhora a manutenção do código e otimiza a escalabilidade de aplicações na AWS.
